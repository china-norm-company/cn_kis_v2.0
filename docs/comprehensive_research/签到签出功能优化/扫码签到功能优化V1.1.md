## 2.1 已确认前提（2026-03-21）

| 项 | 结论 |
|----|------|
| **接待看板 vs 工单执行** | **数据一致**：接待看板展示与工单执行相关接口所用数据对齐，小程序侧按同一套后端事实来源设计展示（以实际调用接口与表为准，不假设双轨不一致）。 |
| **SC 号分配** | **延续现网逻辑**：同一项目下，**访视点为 V1** 且 **签到触发** 时按项目顺序赋号（库内形如 `001/002…`，界面可统一展示为 `SC001`）；**不区分**扫码来自微信小程序还是接待台。 |

## 2.2 登录流程非回归（微信手机号一键登录）

当前 V1.0：**首页 `openType="getPhoneNumber"`** → 将 `code` 提交 **`POST /api/v1/auth/wechat/login`** → 后端 **getuserphonenumber** 换手机号并发 token（见 `backend/apps/identity/api.py` 中 `wechat_login`）。

**V1.1 增量须遵守：**

- **不得**将「获取微信昵称」与「手机号授权」绑为**同一步必填**（例如同一按钮要求两种授权均成功才能登录）。
- **不得**在登录成功前增加**阻塞式**昵称授权；若需昵称，应在 **token 已签发、用户已进入已登录态之后**，以 **可选** 交互获取（用户拒绝则沿用 `name`/「微信用户」/「受试者」等现有展示）。
- 隐私政策 / 用户说明中可补充「展示用途」类表述；**仅改文案不改变接口时**不影响登录能力。

## 附录 A：首页聚合接口与「主项目」规则（可实现规格）

### §1 接口概要

| 项 | 说明 |
|----|------|
| **方法 / 路径** | `GET /api/v1/my/home-dashboard` |
| **鉴权** | 与现有 `/my/*` 一致（手机号 JWT / 兼容 account）；权限建议沿用 **`my.profile.read`**（或与本类只读接口同级的新权限名，需与 `seed_roles` 一致）。 |
| **用途** | 小程序首页一次拉齐：问候用展示名、多项目卡片、入组情况、SC、今日签到态、访视/项目信息，**减少**多次调用 `/my/profile` + `/my/enrollments` + 未来扩展字段。 |
| **Query 参数** | `date`（可选，`YYYY-MM-DD`，默认 **服务端当日** `timezone.localdate()`，与接待台「今日」一致）。 |

**响应**（与全站约定一致：`{ code, msg, data }`；以下描述 **`data`**）：

```json
{
  "as_of_date": "2026-03-21",
  "display_name": "张三",
  "display_name_source": "subject",
  "primary_project": { "project_code": "PRJ01", "...": "见下「项目块」" },
  "other_projects": [ { "...": "项目块" } ],
  "projects_ordered": [ { "...": "项目块，含 is_primary" } ]
}
```

说明：

- **`primary_project`**：折叠态首页用的 **主项目**（可为 `null`）。
- **`other_projects`**：**默认折叠「更多项目」** 时展开列表；不含主项目；**稳定排序**见 §3。
- **`projects_ordered`**（可选但推荐）：主项目 + 其余项目 **同序**，且带 `is_primary: true/false`，前端可只用一个数组渲染。

若希望 **极简响应**，可只返回 `projects_ordered`，由前端取 `is_primary` 区分；上面三项同时给出是为了兼容「先主后副」的直观对接。

### §2 「项目块」字段（每个元素结构一致）

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `project_code` | string | 是 | 项目编号，与预约/导入、`SubjectProjectSC.project_code` 一致 |
| `project_name` | string | 是 | 展示用名称；优先当日/最近预约的 `project_name`，否则 `Protocol.title`，再否则 `project_code` |
| `visit_point` | string | 否 | **as_of_date 当日**关联预约的访视点；无则 `""` |
| `appointment_id` | int \| null | 否 | 用于跳转预约详情等；无则 `null` |
| `enrollment_status` | string | 否 | **`SubjectProjectSC.enrollment_status`** 中文枚举；无记录则 `""` |
| `sc_number` | string | 否 | 库内原始值（如 `001`）；无则 `""` |
| `sc_display` | string | 否 | 展示用 **`SC` + 三位**（如 `sc_number=001` → `SC001`）；无则 `""` |
| `queue_checkin_today` | string | 是 | 与今日队列语义对齐：`none` \| `checked_in` \| `checked_out`（由 **`SubjectCheckin`** 在 `as_of_date` 是否存在、及 `status` 推导，与接待侧约定一致） |
| `enrollment_id` | int \| null | 否 | EDC `Enrollment.id`（若该 `project_code` 能关联到 protocol）；无则 `null` |
| `protocol_id` | int \| null | 否 | 方案 ID，便于跳转 |
| `is_primary` | boolean | 仅 `projects_ordered` | 是否主项目 |

**数据绑定约定（实现侧）**

- **`enrollment_status` / `sc_*`**：来自 **`SubjectProjectSC`**（`subject_id` + `project_code`，`is_deleted=false`）。
- **当日预约**：`SubjectAppointment`，`subject_id` + `appointment_date = as_of_date`，`status` 不含 `cancelled`（与现有列表习惯一致）。
- **队列签到态**：`SubjectCheckin`，`subject_id` + `checkin_date = as_of_date`；若存在多条，取**未签出**优先，否则**最近一条**——需在实现里与现网「今日一条活跃签到」规则一致（与 `quick_checkin` 行为对齐）。

### §3 「主项目」判定规则（可实现、按优先级短路）

对当前受试者，在 **`as_of_date`** 下计算 **唯一** `primary_project`（`project_code`）：

1. **当日存在「进行中签到」**  
   - 条件：`SubjectCheckin` 满足 `checkin_date = as_of_date` 且状态为 **已签到未签出**（与模型里 `CheckinStatus` 一致，实现时用代码枚举比对）。  
   - 动作：取该签到关联的 **当日预约** 的 `project_code`（若模型无直接 FK，则用「当日同一 `subject` 的预约中 `status=COMPLETED` 且 `project_code` 匹配」或业务上已存在的关联方式，**与现网接待队列一致**）。  
   - 若仍无法唯一映射：进入 2。

2. **当日有有效预约**  
   - `SubjectAppointment`：`appointment_date = as_of_date`，非 `cancelled`。  
   - 取 **`appointment_time` 升序第一条**（`appointment_time IS NULL` 的排 **在具体时间之后**，即 **NULLS LAST**）。  
   - 用其 `project_code` 作为主项目。

3. **当日无预约，但有未来预约**  
   - 取 `appointment_date > as_of_date`（或含跨天规则由你们统一）中 **`appointment_date` 升序、`appointment_time` 升序** 第一条的 `project_code`。

4. **仅有 `SubjectProjectSC`、无上述预约**  
   - 取 **`update_time` 最新** 的一条记录的 `project_code`。

5. **仅有 `Enrollment`（如 `enrolled`）**  
   - 取 **`enrolled_at` 最近** 的一条所对应 `protocol.code` 作为 `project_code`（`project_name` 用 `protocol.title`）。

6. **皆无**  
   - `primary_project = null`；`other_projects` / `projects_ordered` 可为空数组。

**`other_projects` 排序**

- 在「全量应展示项目集合」中去掉主项目后：  
  - **按 `project_code` 字典序升序**（实现简单、稳定）；若产品以后要「按下次预约时间排序」，可再改，但需在文档中替换本句。

**「全量应展示项目集合」**（去重合并）

- `project_code` 来自：  
  - `as_of_date` 及 **之后** 非取消预约；  
  - 所有 `SubjectProjectSC`（未删除）；  
  - `Enrollment` 为 `enrolled`（或含 `pending`，按产品：建议 **至少 enrolled + 当日/未来预约**）。  

同一 `project_code` 合并为一条项目块，**名称/访视点**优先取 **as_of_date 当日预约**，否则取最近未来预约，再否则 SC/Enrollment 推导。

### §4 `display_name` 与 `display_name_source`（与 §2.2 一致）

| `display_name` 取值优先级 | `display_name_source` |
|---------------------------|------------------------|
| `Subject.name` 非空且 **不等于** 占位 `"微信用户"` | `subject` |
| 否则，**as_of_date 当日** 第一条有效预约若带受试者展示名（若预约无名字字段则用档案逻辑） | `appointment` |
| 否则，若前端已通过 **登录后可选** 接口把昵称写入后端（若 V1.1 增加 `wechat_nickname` 字段或扩展 profile） | `wechat_nickname` |
| 否则固定 **`受试者`** 或与现网首页一致的 fallback | `fallback` |

说明：**手机号一键登录** 不得依赖昵称；昵称只影响 `display_name` 的 3、4 档，**不阻塞 token**。

### §5 错误与缓存

- **401/403**：与 `/my/profile` 一致。  
- **404**：一般不设（无 subject 时与现网一致返回业务 `code`）。  
- **缓存**：首版可不缓存；若加缓存，TTL 建议 ≤ 30s 或与首页轮询间隔一致。

### §6 与 V1.1 文档的对应关系

- **V1.1-01 / V1.1-02 / V1.1-04 / V1.1-05**：主要由本接口 **`projects_*` + `enrollment_status` + `sc_*` + `queue_checkin_today`** 承载。  
- **V1.1-03**：由 **`display_name` + `display_name_source`** 承载。  
- **默认折叠「更多项目」**：前端用 **`primary_project` + `other_projects`** 或 **`projects_ordered` + `is_primary`**。

## 附录 B：小程序签到/签出与接待看板联动（实现路径）

> **背景**：接待看板「今日队列」使用 `source=board`，数据来自 `ReceptionBoardCheckin` / `ReceptionBoardProjectSc`；小程序 `/api/v1/my/scan-checkin` 成功路径写入 `SubjectCheckin`（工单执行侧）。两套数据默认**互不影响**。本附录固化 **V1.1 推荐实现路径**：在**不改动**小程序对外接口与接待台既有入口的前提下，于小程序成功签到/签出后**镜像**看板数据，使看板状态与小程序一致。

### B.1 目标与非回归约束

| 项 | 说明 |
|----|------|
| **目标** | 受试者通过小程序扫码完成签到/签出后，接待看板（`ReceptionDashboardPage`，`todayQueue` 的 `source: 'board'`）中对应行的签到/签出**状态与时间**与事实一致；看板侧 SC 仍遵循现有 `board_checkin` / `_ensure_board_project_sc_on_checkin` 规则。 |
| **保留** | 小程序仍返回原有 `code/msg/data` 语义；仍调用 `quick_checkin` / `quick_checkout` 维护 **`SubjectCheckin`**（执行台、预约页 `source=execution`、`todayStats` 等**行为不变**）。 |
| **保留** | 接待台看板签到 API（`board-checkin` / `board-checkout`）、接待扫码签到（`/reception/scan-checkin` → `quick_checkin`）**不强制改为双写**；本方案仅约束 **小程序** `/my/scan-checkin` 成功路径的**追加镜像**。 |

### B.2 推荐方案：成功路径追加镜像

在 `backend/apps/subject/api_my.py` 的 **`my_scan_checkin`** 中，在现有逻辑**已成功**后增加：

| 小程序分支 | 已有逻辑 | **追加**（联动看板） |
|------------|----------|----------------------|
| 当日首次签到 | `quick_checkin(subject.id, method='qr_scan', location=location)` | 等价于一次 **`board_checkin`**（`ReceptionBoardCheckin` + 必要时 `ReceptionBoardProjectSc`） |
| 已签到/执行中再扫 | `quick_checkout(existing.id)` | **`board_checkout(subject_id, target_date=today)`**（写看板 `checkout_time`） |

实现建议：

- 在 `reception_service` 中抽取 **`_mirror_reception_board_after_my_scan(subject_id, today, action, checkin_row=None, appt_context=None)`**（名称可调整），**仅**由 `my_scan_checkin` 调用，避免误伤接待台其它接口。
- **不要**修改 `ReceptionDashboardPage` 的 `source: 'board'`；看板仍只读 board 数据源。

### B.3 关键实现细节：`quick_checkin` 与 `board_checkin` 的预约状态顺序

`quick_checkin` 会将当日匹配预约标为 **`COMPLETED`**（`SubjectAppointment`）。现有 **`board_checkin`** 选取预约时仅查询 **`CONFIRMED` / `PENDING`**，若在 `quick_checkin` **之后**再裸调 `board_checkin(subject_id, today)`，可能出现 **`appt` 为空**，导致看板记录缺少 `appointment_id` 或 SC 分配与预期不一致。

**必选其一（实现时在产品评审中确认并写进 PR 说明）：**

1. **带上下文镜像（推荐）**：在 `quick_checkin` 内将预约改为 `COMPLETED` **之前**，保存当次使用的 `appointment_id`、`project_code`、`visit_point`；镜像函数用该上下文写入/更新 `ReceptionBoardCheckin`，并调用与 `board_checkin` 同款的 **`_ensure_board_project_sc_on_checkin`**（可将「给定 appt 写看板」抽为内部函数，避免依赖再次查询 `CONFIRMED/PENDING`）。
2. **扩展 `board_checkin`（增量、向后兼容）**：当无 `CONFIRMED/PENDING` 时，**回退**查询当日 **`COMPLETED`** 预约（如按 `update_time` 最近）用于绑定 `appointment_id` 与 SC。需覆盖「仅看板签到」「小程序后再看板」等回归场景。

### B.4 事务与失败策略

- 小程序单次请求内，**执行侧写入**与**看板镜像**宜置于**同一数据库事务**（`@transaction.atomic`），避免出现小程序提示成功而看板未更新。
- **镜像失败**：应**整体回滚**本次签到/签出（含 `quick_checkin`/`quick_checkout` 已写内容），或按重试策略处理；**禁止**静默吞异常导致长期双轨分裂。
- **签出镜像**：`board_checkout` 在当日**无** `ReceptionBoardCheckin` 时会抛错。若签到镜像已成功则通常存在看板记录；若签到镜像曾失败，签出镜像需 **catch** 后与会话状态一致处理（一般与事务回滚策略统一）。

### B.5 SC 与「数据联动」边界

- **执行侧 SC**：`SubjectProjectSC`（`quick_checkin` 已维护）。
- **看板侧 SC**：`ReceptionBoardProjectSc`（镜像走看板既有逻辑）。

本方案实现的是**看板队列状态与时间**与小程序一致；**两套 SC 表仍独立**。若产品要求看板与执行侧 **SC 号码严格一致**，需另增「拷贝/对齐」子需求，不在本附录默认范围内。

### B.6 测试与回归清单（建议）

- 小程序：首次签到、已签出再扫、过期/无效场所码（与 V1.0 一致）。
- 看板：仅看板签到/签出、小程序签到后看板行状态、小程序签出后看板签出时间。
