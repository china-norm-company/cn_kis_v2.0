## 一、目标与边界

### 1.1 业务目标

在**统一后端**之上，完成受试者**微信小程序日记**与**采苓·研究台**、**维周·执行台**的配置、触发、填报、落库、查询/下载/督导管理的**端到端闭环**，满足可追溯与监查导出需求。

### 1.2 范围边界（须遵守）

| 事项 | 说明 |
|------|------|
| **工作台归属** | 日记配置与数据下载属 **采苓·研究台**；督导确认、质疑打回与执行侧管理属 **维周·执行台**；不得合并或串台。完整约束见 `docs/WORKSTATION_INDEPENDENCE.md`。 |
| **确认门禁分阶段** | **2.0**：配置生效前的**研究员确认**（研究台侧或后端等效记录），**不要求**执行台督导参与。**3.0**：可选「督导在执行台二次确认」及质疑打回等完整执行台能力。详见 [`小程序日记功能2.0.md`](./小程序日记功能2.0.md) 文首版本口径表。 |
| **1.0 已交付** | 小程序日记页基础能力、提交形态、不扩大首页/其他页面范围等，以《小程序日记功能 1.0》文档为准。 |
| **2.0 分工重点** | **小程序 ↔ 后端 ↔ 研究台侧数据链路**；首包配置可在**后端直接模拟研究台下发的落库**（脚本/管理命令/内部 API，见 [`小程序日记功能2.0.md`](./小程序日记功能2.0.md) §四），**不依赖**采苓 R3/R4 页面；研究台「完整配置产品化界面」由同事交付后对接**同一套写入逻辑**。 |

## 二、版本总览（1.0 → 4.0）

| 版本 | 主题 | 核心交付 | 验收要点 |
|------|------|----------|----------|
| **1.0** | 小程序端基础 | 日记界面、入口、端内提交与既有接口契约下的可用性 | 与《小程序日记功能 1.0》一致；**已完成** |
| **2.0** | **研究台方向数据链** | **任务触发（配置下发）**、**填报数据在研究台可见/可下载**；配置**在后端模拟研究台落库**（见 2.0 文档 §四） | 端到端真数据；采苓 R3/R4 完整配置页面可后置 |
| **3.0** | **执行台方向能力** | 督导**可选二次确认配置**、**质疑打回**、**进度/提醒/管理**等与后端打通 | 与 [**主链图**](./小程序日记开发整体需求_图.md) 中执行台侧步骤一致；**2.0 不含**督导确认与打回 |
| **4.0** | **全闭环 + 体验与合规细化** | 全角色、全链路、提醒/审计/导出/权限与边界情况 | 对齐 [**主链图**](./小程序日记开发整体需求_图.md) 六步；可发布级质量 |

## 三、分版本需求说明

### 3.1 1.0（基线，已完成）

**范围**：微信小程序 `pages/diary` 及相关共享包；**不**擅自扩展后端契约（以 1.0 文档为准）。

**验收**：功能列表、不改动范围、入口与提交字段约定与《小程序日记功能 1.0》一致。

---

### 3.2 2.0 — 小程序与研究台数据传输（当前主轨）

**目标**：实现「研究台侧能**驱动**受试者侧任务感知」与「研究台侧能**拉取/导出**真实填报数据」，中间经**真实后端**与数据库。

**约束**：

- 研究台「复杂配置产品化**界面**」若由同事交付：**首包在后端模拟研究台配置落库**（固定项目/协议、种子、`manage.py`、内部 API 等，见 [`小程序日记功能2.0.md`](./小程序日记功能2.0.md) §四），使 **B1/B2 类能力在数据层生效**；页面上线后调用同一套接口即可。
- **必须真传输**：小程序调用的接口、研究台列表/下载调用的接口、落库记录一致，可演示「配置变更 → 小程序侧行为变化 → 研究台看到同一条目」。
- **2.0 不含**：维周·执行台侧「督导确认配置」「质疑打回」等流程；配置**上线确认**由**研究员**完成（见 [`小程序日记功能2.0.md`](./小程序日记功能2.0.md)）。

**建议需求条目（可映射《小程序日记功能 2.0》中 F/R/B）**：

| 模块 | 需求摘要 | 说明 |
|------|----------|------|
| **后端** | 日记配置存储与下发；日记条目 CRUD；列表与导出接口；鉴权与项目维度 | 对齐 2.0 文档 B1～B4；未完成汇总 B5 若纳入本阶段则一并验收 |
| **小程序** | 拉取「今日/待填」与表单相关元数据；提交与补填字段与后端一致；错误与时间窗提示 | 对齐 F1、F8 等与配置下发相关的项 |
| **研究台（最小）** | 列表查询 + 导出；若 UI 未就绪，可用**同等接口的本地调用**证明数据 | 对齐 R1、R2；R3、R4 **非首包前置**，配置由 **§四 后端模拟** 写入 |
| **后端模拟配置** | 文档化：如何在**后端**写入与「研究台已下发」等效的**生效配置** | 见 [`小程序日记功能2.0.md`](./小程序日记功能2.0.md) §四；建议另建《本地跑通手册》 |

**2.0 验收标准（建议）**：

1. 可重复步骤：写入配置 → 小程序拉取结果可验证变化 → 提交若干条日记 → 研究台侧接口（或页面）可列表、可导出，**字段含日记日期、提交时间、补填标识等**（与 2.0 补填/合规表一致）。
2. 不以仅前端静态数据冒充后端贯通；联调前占位须在文档与代码中**明确标注**。
3. 更新本文件与 [`小程序日记开发整体需求_图.md`](./小程序日记开发整体需求_图.md) 中关于 **2.0 与模拟配置边界** 的表述（若主链表意有变）。

---

### 3.3 3.0 — 执行台打通

**目标**：[**主链图**](./小程序日记开发整体需求_图.md) 中 **执行台·督导** 侧与后端一致：第 3 步（督导确认配置）、第 6 步（督导管理）等。**说明**：**2.0** 阶段第 2 步的「确认」仅为**研究员确认**；图中执行台第 3 步为 **3.0** 建设范围，2.0 联调可不依赖该分支。

**需求要点（实现路径与路由见工程，图中不写）**：

- **可选**督导二次确认：由研究员/项目在策略中决定是否必须经过督导；督导查看配置并确认时，确认记录写入后端，与配置版本关联。
- **质疑打回**：部署在执行台；受试者条目提交后不可随意修改，仅打回后可改（与 [`小程序日记功能2.0.md`](./小程序日记功能2.0.md) §七 一致）。
- 进度汇总、待人工提醒/跟进清单等与后端汇总能力一致；可与消息渠道同源。
- 权限：督导/PM 等角色可写确认及管理类写操作（具体以 RBAC 为准）。

**验收**：

- 与 2.0 **同一套库与数据模型**；若产品规则要求「未督导确认不可见」，确认状态须能约束小程序可见配置。
- 与研究台下载使用**同一条目事实源**。

---

### 3.4 4.0 — 全闭环与优化

**目标**：产品级完整日记闭环 + 体验与合规细化。

**建议包含**：

| 类别 | 内容 |
|------|------|
| **功能完整** | 六步主链无断点；双入口（研究/执行）+ 小程序 + 后端全接通 |
| **合规与审计** | 补填原因、日记日期与实际提交时间、修改审计、导出字段完备（见《小程序日记功能 2.0》补填小节） |
| **运维与质量** | 与发布门禁、UAT 模板一致；监控与回滚预案 |
| **体验** | 弱网、超时、空状态、未配置、未（研究员/督导）确认、时间窗外等边界 |

**验收**：发布门禁通过；本文件 1.0～4.0 **里程碑可勾选关闭**；与 `WECHAT_MINI_INCOMPLETE_AND_OPTIMIZATION.md` 无冲突或可记录例外。
