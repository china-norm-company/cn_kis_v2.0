# PQ — 性能确认（Performance Qualification）

**系统**：CN-KIS V2.0  
**文件编号**：CN-KIS-CSV-PQ-001  
**版本**：0.1（草稿）  
**状态**：待执行

---

## 1. 目的

验证 CN-KIS V2.0 在真实业务场景下能够达到 FRS 中定义的性能、数据完整性和业务流程完整性要求。

---

## 2. 可接受标准（Acceptance Criteria）

| 编号 | 场景 | 指标 | 可接受标准 |
|------|------|------|----------|
| PQ-AC-001 | API 响应性能 | P95 响应时间 | < 500ms（非 AI 接口）|
| PQ-AC-002 | 知识检索性能 | 语义检索 P95 延迟 | < 2000ms |
| PQ-AC-003 | 并发用户数 | 同时在线 50 用户下响应时间 | < 1s P95 |
| PQ-AC-004 | 数据审计完整性 | 业务操作与审计日志的匹配率 | 100%（0 丢失）|
| PQ-AC-005 | 备份恢复 | 数据库备份恢复时间 | RTO ≤ 4h, RPO ≤ 24h |
| PQ-AC-006 | 知识写保护 | 写保护状态下触发写入的拦截率 | 100% |

---

## 3. 测试场景

### PQ-001：端到端 CRF 录入流程

**目的**：验证完整的 CRF 数据录入链路符合 GCP 要求

| 步骤 | 操作 | 期望结果 | 接受标准 |
|------|------|---------|---------|
| 1 | 创建研究方案 V1.0 | 方案创建成功，`t_protocol.version='1.0'` | ✅ 记录版本 |
| 2 | 注册受试者并生成假名编码 | `t_subject_pseudonym` 创建，`pseudonym_code` 生成 | ✅ 假名化成功 |
| 3 | 录入 eCRF 数据（10 条） | 10 条 `t_crf_record` 创建，10 条 `t_audit_log` 记录 | ✅ 100% 审计覆盖 |
| 4 | 查询审计日志 | 返回全部 10+ 条相关记录 | ✅ 0 条丢失 |
| 5 | 尝试删除审计日志 | HTTP 403 拒绝 | ✅ 已拦截 |

### PQ-002：知识资产生命周期

| 步骤 | 操作 | 期望结果 | 接受标准 |
|------|------|---------|---------|
| 1 | 创建知识条目（状态=draft）| 创建成功，`index_status=pending` | ✅ |
| 2 | 触发向量化 | 向量化完成，`index_status=indexed` | ✅ ≤ 60s |
| 3 | 语义检索包含新条目 | 检索结果命中 | ✅ |
| 4 | 运行质量巡检 | 无 `no_title` 告警 | ✅ |
| 5 | 设置 `next_review_at` = 过去时间 | Celery 任务次日推送飞书提醒 | ✅ |

### PQ-003：并发访问性能

| 测试工具 | 配置 | 指标 | 目标 |
|---------|------|------|------|
| `locust` 或 `wrk` | 50 并发用户 / 60s 持续 | P95 延迟 | < 1000ms |
| 同上 | 冷启动首次登录 | 认证完成时间 | < 3s |
| 同上 | 知识检索（语义） | P95 延迟 | < 2000ms |

