## 1. 背景与业务目标

### 1.1 要解决什么

- **研究侧**：按「全链路项目」维护一份**电子日记配置**（题目、填写规则、应填周期、时间窗、补填回溯天数等）；配置需**发布**且**研究员确认**后，受试者小程序才能拉到任务。
- **受试侧**：在微信小程序中查看任务说明、按「规定使用日期」填写或补填日记；支持症状等字段的**结构化拆分**；同一自然日内对**同一受试者**仅允许**一条**日记（防重复）。
- **数据一致**：研究台列表里的「是否发生任何不良情况」与小程序展示语义一致（见第 5 节判定规则）。

### 1.2 本版刻意不做的能力（避免合并时误预期）

- **同一受试者、同一自然日、分属两个全链路项目各记一条日记**：当前库表约束为「受试者 + 日期」唯一，**未**增加 `project_id` 维度；若主线或其它分支要支持「多项目同日多日记」，需要**另行迁移与接口改造**，与本次 2.0 **不属于同一交付**。

## 2. 数据模型（后端）

### 2.1 `SubjectDiaryConfig`（日记配置，项目级）

- 绑定全链路 **`project_id`**（及冗余 **`project_no`** 等展示字段，具体见模型）。
- **状态**：草稿 / 发布等；受试端只认「已发布且 **researcher_confirmed_at 非空**」的最新配置。
- **`form_definition_json`**：表单题目（如是否用药等）。  
- **`rule_json`**：规则（如 `diary_period` 起止、`fill_time_window`、`timezone`、`retrospective_days_max` 等）。

### 2.2 `SubjectDiary`（日记条目）

- **`subject_id` + `entry_date`**：数据库**唯一**（`unique_together`），即每人每天最多一条物理记录。  
- **字段**：`mood`，`symptoms`，`medication_taken`，`symptom_severity`，`symptom_onset`，`symptom_duration`，`notes`，`is_deleted` 等；**无 `project_id` 列**（故无法按项目拆同日多条）。  
- **`entry_date`**：业务上的「规定使用日期」，可与提交日不同（补填场景）。

### 2.3 文本规范化（`diary_text` 模块）

- 入参中 `mood` / `symptoms` / `notes` / 各症状拆分字段经 **`normalize_diary_text_field`** 再入库，避免对象被写成 `[object Object]`。  
- 历史数据若把程度/开始/时长拼在 `notes` 里，可通过 **`split_legacy_concatenated_notes`** 解析，与列表里的拆分列对齐。

## 3. 受试端 API（统一前缀 `/api/v1/my/`）

所有路径均挂在现有 Subject「我的」路由下；鉴权为受试者身份及权限 `my.profile.read` / `my.profile.update` 等（与现有 `api_my` 一致）。

| 方法 | 路径 | 权限（典型） | 作用 |
|------|------|--------------|------|
| GET | `/diary/config` | read | 拉取日记配置。Query 可选 **`project_id`**（全链路项目主键）。不传时由服务端 **`_resolve_diary_project_id_for_subject`** 在多个候选中解析**一个**「有可用已发布已确认配置」的项目；解析失败返回 404 及可读 `msg`。 |
| GET | `/diary` | read | 分页返回当前受试者**全部**未删除日记行（**不按 `project_id` 过滤 `items`**）。Query 可选 **`project_id`**：仅用于计算响应里的 **`diary_period`、`retrospective_days_max`** 等元数据，使之与所选项目配置对齐。 |
| POST | `/diary` | update | 新建日记。Body 为 **`DiaryEntryIn`**：`mood`，`symptoms`，`medication_taken`，`symptom_severity`，`symptom_onset`，`symptom_duration`，`notes`，**`entry_date`（可选，YYYY-MM-DD）**。若该 `subject_id + entry_date` 已存在未删除记录 → **400**，提示不可重复提交。 |

**说明**：小程序可通过路由参数或环境变量传入 **`project_id`**，使「配置 + 列表元数据」对齐同一项目；即便如此，**列表 `items` 仍为该受试者全局日记**，合并其它分支时若改为「按项目过滤列表」，属于行为变更需单独评审。

## 4. 研究台 API（统一前缀 `/api/v1/research/diary/`）

需研究侧权限（如 `subject.subject.read` / `subject.subject.update`，以路由装饰器为准）。

| 方法 | 路径 | 作用摘要 |
|------|------|----------|
| POST | `/setup-project` | 日记场景：新建全链路项目并创建与 **project_no 一致**的研究协议编号，便于入组与配置匹配。 |
| POST | `/register-subject` | 按项目登记受试者并入组（日记验证/试点）。 |
| GET | `/linked-protocol` | 按 `project_id` 解析关联的研究协议 ID。 |
| GET | `/project-subjects` | 列出某项目下入组受试者（脱敏手机号等）。 |
| GET | `/configs` | 按项目列出日记配置。 |
| POST | `/configs` | 新建配置。 |
| GET | `/configs/{config_id}` | 配置详情。 |
| PUT | `/configs/{config_id}` | 更新配置。 |
| POST | `/configs/from-template` | 从模板项目复制草稿配置。 |
| POST | `/configs/{config_id}/publish` | 发布。 |
| POST | `/configs/{config_id}/confirm-researcher` | **研究员确认**（2.0 门禁，与受试端拉配置条件一致）。 |
| POST | `/configs/{config_id}/draft` | 撤回草稿。 |
| GET | `/entries` | 工作台日记条目分页列表；可选 **`subject_id`** 筛选。 |
| GET | `/entries/export` | 导出 CSV（UTF-8 BOM），见第 6 节。 |
| GET | `/progress` | 按配置周期与受试者计算应填进度。 |

## 5.「不良情况」判定（研究台 `has_adverse` 与小程序一致）

以下任一为真则视为「有不良」：

1. 规范化后的 **`mood`** 等于 **`不适`**；或  
2. 规范化后的 **`symptoms`** 非空；或  
3. 拆分字段 **`symptom_severity` / `symptom_onset` / `symptom_duration`** 经 **`diary_symptom_fields_for_api`**（含从旧 `notes` 解析）后**任一字段非空**。

研究台列表字段名 **`has_adverse`**（布尔）；导出 CSV 中为 **是/否**。  
小程序 **`subject-core`** 中 **`diaryHasAdverseFromMyItem`** 与上述语义对齐，用于详情文案，避免仅看 `symptoms` 造成与表格矛盾。

## 6. 研究台「数据查看」表格与 CSV 导出

### 6.1 表格列（与页面一致，顺序供对照）

从左到右包含：**受试者编号**、**姓名**、**手机号**、**规定使用日期**、**提交时间**、**是否按要求使用产品**、**是否发生任何不良情况**、**症状**、**症状程度**、**症状开始时间**、**症状持续时长**等（空文本类列界面可为 「—」）。

### 6.2 导出 CSV

- **编码**：UTF-8 **带 BOM**，便于 Excel 打开。  
- **表头**：中文；列顺序在「姓名」之后含 **手机号**，整体与表格语义一致，并在最前增加 **项目编号**、**项目名称**。  
- **项目列**：优先与受试端「日记任务所属项目」解析逻辑一致；无法命中时再尝试「已入组协议编号对应全链路项目」；仍无则 **`—`**。  
- **布尔**：**是 / 否**（不使用 0/1）。  
- **提交时间**：**服务器本地时区** `YYYY-MM-DD HH:mm:ss`（与前端格式对齐）。  
- **症状类空值**：**`—`**；换行替换为空格。  
- **体量**：单次导出最多 **5000** 条（与实现一致）。

## 7. 小程序与共享包（前端）

### 7.1 小程序（`apps/wechat-mini`）

- **日记页**：加载配置、填写前说明、补填说明、表单校验、提交确认、历史与详情、下拉刷新；针对首屏白屏等做过 **ScrollView / 背景色** 等适配。  
- **项目维度**：支持 URL / 编译参数 **`project_id`** 及 **`TARO_APP_DIARY_PROJECT_ID`**，联动配置与列表元数据。  
- **首页**：展示「待填日记」相关提示/角标（依赖受试端列表或同类接口能力）。  
- **详情**：「不良情况」使用 **`diaryHasAdverseFromMyItem`**。

### 7.2 `packages/subject-core`

- **`MyDiaryEntryItem`**、配置类型与 **`GET /my/diary`、`GET /my/diary/config`** 字段对齐。  
- **`useDiaryList(api, projectId)`**：向 **`GET /my/diary`** 传递 `project_id`（大于 0 时）。  
- **`useDiarySubmit`**：提交体类型包含 **`symptom_severity` / `symptom_onset` / `symptom_duration`**，与 **`DiaryEntryIn`** 一致。  
- **`diaryHasAdverseFromMyItem`**：与第 5 节规则一致。

**构建注意**：`subject-core` 需执行 `pnpm --filter @cn-kis/subject-core build`（或仓库既定命令）生成 `dist/`，小程序工程依赖其产物。

## 8. 身份与其它耦合（本节仍属同一分支交付上下文）

日记联调依赖受试者能打通 **小程序登录 → Subject 记录 → `my.profile.*`**。同一分支上常见关联改动包括：**微信登录后落库/关联 `t_subject`**、**短信登录权限**等；合并时请一并关注 **`apps/identity`**、`api_my` 权限装饰器及相关迁移，避免出现「能登录但不能拉日记」的权限断裂。

### 9.2 合并后最小回归（约 10～15 分钟）

1. 研究台：对测试项目 **发布并确认**日记配置 → 小程序能拉到配置与规则。  
2. 小程序：**当日首次提交成功**；**同日同 entry_date 再次提交** → 接口 **400**，提示合理。  
3. 研究台列表与 **导出 CSV**：列含义一致；**不良情况**与症状列无逻辑矛盾；**项目编号/名称、手机号**合理（无则 `—`）。  
4. 小程序日记详情「不良情况」与列表/研究台一致。  
5. `pnpm --filter @cn-kis/subject-core build` 通过。
