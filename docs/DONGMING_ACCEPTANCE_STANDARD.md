# 洞明工作台验收标准

> 适用对象：`洞明·数据台（data-platform）`
> 依据：系统设计规划、现有实现、`docs/DONGMING_WORKSTATION_USER_GUIDE.md`

---

## 1. 验收目标

验收的目标不是证明“页面能打开”，而是证明洞明已经满足以下要求：

- 作为独立数据治理工作台可稳定登录
- 能按数据域、生命周期、追溯链组织能力
- 能观察外部接入、知识治理、分类分级、同步、存储、备份、拓扑等关键治理对象
- 页面与 API 响应一致、无明显硬编码漂移
- 具备可持续测试和回归验证基础

---

## 2. 验收分级

### P0

必须通过；任一失败视为当前版本不可验收：

- 独立 OAuth / 权限门禁
- 核心路由可用
- 治理驾驶舱可用
- 外部接入治理链路可见
- 数据目录 / 分类分级 / 知识治理核心接口可用

### P1

重要功能；允许个别缺陷，但必须有明确说明与补救计划：

- 生命周期、原始来源、知识来源、内容入库、血缘、同步、存储、备份、拓扑
- API 与页面数据一致性
- 主要提示信息、空态、异常态

### P2

增强项；不阻断验收，但需纳入后续优化：

- 视觉细节
- 图谱交互增强
- 更深层对象级追溯
- 辅助文案与提示优化

---

## 3. 总体验收门槛

版本通过验收需满足：

- 全部 `P0` 通过
- `P1` 通过率不低于 `90%`
- 不存在阻断性白屏、死循环登录、关键接口 500
- 已形成可复用测试证据：日志、截图、报告

## 5. 数据一致性验收标准

需要验证以下一致性：

- 页面统计与 `data-platform` API 返回一致
- `数据目录` 的表信息与后端模型/Schema 一致
- `知识转化` 页面与 `knowledge-governance/transformation` 一致
- `治理驾驶舱` 聚合数据与对应子页面口径一致

不通过表现：

- 页面显示旧字段
- 页面仍用硬编码常量
- 新 API 存在但页面未消费

---

## 9. 不通过判定

出现以下任一情况，版本判定为不通过：

- `20029` / 登录循环 / OAuth 串台
- Dashboard、Catalog、Knowledge、Classification 任一关键页面不可用
- 核心 `data-platform` API 大量 500/404
- 页面与实际实现严重漂移，说明文档无法指导使用
