# 洞明工作台全场景系统使用说明与功能说明

> 版本：v1.0
> 适用范围：`洞明·数据台（data-platform）`
> 基线来源：`docs/DONGMING_DATA_PLATFORM_PANORAMA.md`、既有重构规划、当前前后端实现

---

## 1. 文档目的

本文档用于统一说明洞明工作台的：

- 系统定位与边界
- 角色与使用前提
- 页面导航与功能说明
- 关键业务场景与操作路径
- 常见问题与异常处理

洞明的核心定位不是业务执行台，也不是经营分析台，而是**数据治理工作台**。它回答的是“数据是否被专业管理”，而不是“业务做得如何”。

---

## 2. 系统定位与边界

### 2.1 定位

洞明围绕三条主线组织能力：

- `数据域`：系统有哪些数据域，各域边界、规模、责任人与合规状态是什么
- `生命周期`：数据处于 raw / staging / formal / content / knowledge / meta 哪一层，是否按规定流转
- `追溯链`：对象从来源、接入、转化、知识化到检索的链路是否可追溯

### 2.2 服务对象

主要角色：

- `data_manager`：关注数据域健康、接入治理、知识治理、治理缺口
- `tech_director`：关注系统拓扑、存储、同步、备份、演进能力
- `admin` / `superadmin`：关注写保护、恢复、环境配置、发布门禁
- `data_analyst`：关注可用数据域、知识覆盖、检索与转化效果

### 2.3 不负责的事情

洞明不负责：

- 受试者执行进度管理
- 财务经营结果分析
- 质量整改业务闭环
- CRM 机会推进

以上属于专业业务工作台；洞明只负责这些业务数据的治理可见性。

## 3. 使用前提

### 3.1 访问入口

- 生产/测试入口：`/data-platform/`
- 本地开发入口：`http://localhost:3020/data-platform/`

### 3.2 认证前提

洞明使用独立飞书应用：

- App ID：`cli_a93753da2c381cef`

OAuth 成功的前提：

- 前端授权 URL 使用的 `redirect_uri`
- 飞书开放平台登记的重定向地址
- 后端换票使用的 `redirect_uri`

三者必须**完全一致**，包括：

- 协议：`http` / `https`
- 域名或 IP
- 路径：`/data-platform/`
- 尾部斜杠

### 3.3 常见认证错误

`错误码 20029` 通常表示授权页 `redirect_uri` 未在飞书后台登记。

排查顺序：

1. 确认当前访问入口是 `china-norm.com`、`118.196.64.48` 还是 `localhost:3020`
2. 确认前端 App ID 为 `cli_a93753da2c381cef`
3. 确认飞书后台登记了完全一致的 `redirect_uri`
4. 确认后端环境变量 `FEISHU_REDIRECT_BASE` 与前端访问域一致

## 4. 导航与页面说明

### 4.1 一级导航结构

| 一级分组 | 页面 |
|---|---|
| 治理驾驶舱 | 治理驾驶舱、数据域地图、数据生命周期 |
| 外部数据治理 | 候选接入池、原始来源 |
| 知识治理 | 知识条目、知识来源、内容入库 |
| 分类与合规 | 数据目录、分类分级、数据质量 |
| 血缘与追溯 | 数据血缘图谱 |
| 同步与存储 | 同步管道、存储容量、备份状态、服务拓扑 |

### 4.2 页面级功能说明

#### 4.2.1 `治理驾驶舱`

目的：

- 展示跨域治理健康度
- 汇总治理缺口、生命周期滞留、向量化进度、服务拓扑

核心信息：

- 治理缺口数量与严重度
- 待审核接入量
- 各域数据规模
- 知识向量化覆盖率
- 分类分级合规摘要

适合回答：

- 哪些数据域当前有风险
- 哪些治理动作必须优先处理
- 系统治理是否处于可控状态

#### 4.2.2 `数据域地图`

目的：

- 用数据域而不是业务模块来理解系统数据版图

核心信息：

- 数据域定义
- 域归属表
- 生命周期阶段
- 责任角色
- 合规标签

适合回答：

- 某类数据属于哪个域
- 某域由谁负责
- 某域承载在哪些表上

#### 4.2.3 `数据生命周期`

目的：

- 从 `raw -> staging -> formal -> content -> knowledge -> meta` 观察数据流转

核心信息：

- 各层数据量
- 各域在各层的分布
- 滞留对象和积压对象

适合回答：

- 数据卡在什么阶段
- 是否存在长期未接入、未转化、未向量化对象

#### 4.2.4 `候选接入池`

目的：

- 管理外部数据从原始层进入正式域前的候选、审核与接入

核心信息：

- 候选总量、待审核量、高置信待审量
- 各来源类型分布
- 各专业工作台承接情况

关键动作：

- 全局生成候选
- 查看来源分布
- 跟踪各域承接情况

#### 4.2.5 `原始来源`

目的：

- 统一治理 LIMS、易快报、飞书等外部原始来源

核心信息：

- 原始记录规模
- 冲突汇总
- 来源分布
- 候选与接入状态

适合回答：

- 哪个外部来源断流或堆积
- 哪类冲突最常见

#### 4.2.6 `知识条目`

目的：

- 管理知识资产、转化质量和复核状态

核心信息：

- 条目列表
- 质量分
- 状态
- 复核到期
- 向量化覆盖
- 按来源类型转化率

关键动作：

- 按类型筛选知识
- 搜索标题/摘要
- 查看逾期复核条目

#### 4.2.7 `知识来源`

目的：

- 展示各类知识来源注册表与状态

核心信息：

- 来源类型
- 负责人
- 调度方式
- 最近抓取时间
- 状态

#### 4.2.8 `内容入库`

目的：

- 管理 PersonalContext 到 KnowledgeEntry 的入库、去重、向量化过程

核心信息：

- 原始来源分布
- 重复组
- 待处理条目
- dry-run 与实际执行结果

关键动作：

- 去重清洗
- 触发入库 pipeline
- 观察待向量化状态

#### 4.2.9 `数据目录`

目的：

- 用“目录 + Schema + 分类”统一查看核心表

核心信息：

- 核心表列表
- 字段结构
- 记录数
- 不可变标识
- 向量化标识
- 责任人 / 保留期 / 合规标签

适合回答：

- 某张表是否存在、字段是什么、归属哪个治理层

#### 4.2.10 `分类分级`

目的：

- 展示六维分类体系

维度：

- `security_level`
- `criticality`
- `regulatory_categories`
- `freshness_sla`
- `retention_years`
- `data_owner_role`

关键用途：

- 做合规检查
- 识别 GCP + PIPL 冲突
- 跟踪待假名化对象

#### 4.2.11 `数据质量`

目的：

- 展示质量规则与告警

核心信息：

- 规则列表
- 告警列表
- 健康状态
- 与知识写保护、上下文规模等治理项联动

#### 4.2.12 `数据血缘图谱`

目的：

- 展示静态数据流、知识图谱、方案版本血缘

核心视图：

- 核心流转血缘
- 真实知识图谱
- 方案版本血缘

适合回答：

- 某类数据从哪里来、往哪里去
- 知识图谱中实体如何连接

#### 4.2.13 `同步管道`

目的：

- 展示 Celery Beat 与关键同步任务调度状态

核心信息：

- 任务名
- 调度周期
- 类别
- 是否启用

#### 4.2.14 `存储容量`

目的：

- 展示 PostgreSQL、Redis、Qdrant 等存储状态与规模

核心信息：

- 数据库大小
- 表规模
- Redis 占用
- Qdrant collection

#### 4.2.15 `备份状态`

目的：

- 展示备份文件发现状态与恢复准备度

核心信息：

- 最近备份时间
- 文件数量
- age_hours
- 是否 stale/no_backup

#### 4.2.16 `服务拓扑`

目的：

- 从运行态观察 Postgres、Redis、Qdrant、Celery Broker 等基础组件

核心信息：

- healthy / degraded
- 队列深度
- 错误摘要

## 5. 全场景使用路径

### 场景 A：发现治理风险

1. 进入 `治理驾驶舱`
2. 查看治理缺口和生命周期滞留
3. 进入对应详情页（如原始来源、内容入库、分类分级）
4. 判断是接入问题、质量问题、合规问题还是存储问题

### 场景 B：核对某个数据域

1. 进入 `数据域地图`
2. 选择目标域
3. 查看该域包含的表、生命周期阶段、负责人与合规标签
4. 必要时到 `数据目录` 查看字段和行数

### 场景 C：处理外部接入堆积

1. 进入 `候选接入池`
2. 查看待审核总量和高置信待审量
3. 到 `原始来源` 识别具体来源系统
4. 到 `数据生命周期` 确认是否卡在 staging

### 场景 D：检查知识沉淀效果

1. 进入 `知识条目`
2. 观察向量化覆盖率、质量分布、来源转化率
3. 到 `内容入库` 查看 pending / failed
4. 到 `治理驾驶舱` 复核整体缺口

### 场景 E：做合规复核

1. 进入 `分类分级`
2. 查看 SEC-4、GCP、PIPL、冲突项
3. 到 `数据目录` 确认具体表
4. 需要时到 `备份状态` / `服务拓扑` 复核基础保障

### 场景 F：做追溯与问题定位

1. 进入 `数据血缘图谱`
2. 观察来源、存储、知识化链路
3. 必要时结合候选/上下文 trace API 做对象级追溯

## 6. 数据与权限说明

### 6.1 数据域分类

当前洞明重点覆盖的数据域包括：

- 外部原始数据域
- 接入暂存域
- 业务正式数据域
- 内容与信号域
- 知识资产域
- 治理元数据域

### 6.2 权限特征

洞明中的大多数页面是只读治理页；高风险操作主要是：

- 候选生成
- 去重执行
- 触发入库 pipeline
- 知识写保护相关动作

这些操作必须受权限控制并保留审计痕迹。

## 7. 常见问题

### 7.1 登录时报 `20029`

说明 `redirect_uri` 不匹配。优先检查：

- 当前访问入口
- 飞书后台登记项
- `FEISHU_REDIRECT_BASE`
- `VITE_FEISHU_APP_ID`

### 7.2 页面空白或回到登录页

优先检查：

- JWT / OAuth 是否失效
- `/auth/profile` 是否成功
- 当前账号是否具备 `data-platform` 可见权限

### 7.3 数据看起来“不对”

优先检查：

- 页面是否读取实时 API 还是静态常量
- 数据是否卡在 raw / staging / pending vectorization
- 上游同步任务是否异常
