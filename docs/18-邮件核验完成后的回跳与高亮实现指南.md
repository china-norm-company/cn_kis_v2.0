# 执行台：邮件核验完成后的回跳与高亮实现指南

> **目的**：说明本仓库中「双签相关 **2 封邮件**」在用户完成操作后，如何**回到知情管理列表 / 双签工作人员名单**并**高亮对应数据**；供其它项目对照实现，并排查「总是回到执行台首页」类问题。  
> **版本**：1.0  
> **范围**：执行台 SPA（`HashRouter`）、后端邮件链接、`#/witness-verify` 核验页。

---

## 1. 两封邮件分别是什么

| 邮件 | 后端入口（概念） | 用户动作 | 完成后应去哪 |
|------|------------------|----------|--------------|
| **A. 项目双签授权** | `send_witness_authorization_email`（绑定 `protocol_id` + `icf_version_id`） | 人脸核验 → **同意/拒绝授权** | **知情管理 · 项目列表**，并**高亮该项目行** |
| **B. 双签档案核验（名单）** | `send_witness_profile_verification_email`（不绑定协议） | 人脸核验 → **手写签名登记** | **双签工作人员名单**，并**高亮该档案行** |

两封邮件里的**首跳链接形式相同**，都是进入独立全屏页（无侧栏）：

```text
{执行台公网基址}/#/witness-verify?token={一次性令牌}
```

基址由后端 `_public_execution_base_url()` 生成，读取环境变量 **`EXECUTION_PUBLIC_BASE_URL`** 或 **`FRONTEND_EXECUTION_URL`**（见 `backend/settings.py`），缺省开发环境常为 `http://127.0.0.1:3007`。  
邮件正文内链接：`backend/apps/protocol/services/witness_staff_service.py` 中 `link = f'{base}/#/witness-verify?token={token_plain}'`。

## 2. 核验页如何区分「回列表」还是「回名单」

**页面**：`apps/execution/src/pages/WitnessFaceVerifyPage.tsx`。

1. 前端用 `token` 调 **`witness-auth/resolve`**（以 `packages/api-client` 实际路径为准），解析出：
   - `token_scope`: **`project`**（项目授权邮件）或 **`profile`**（名单核验邮件）
   - `protocol_id`：仅 **project** 有意义
   - `witness_staff_id`：用于名单高亮

2. **完成态 UI** 上的主按钮使用 React Router **`Link`**（不是裸 `<a href>` 整页刷新），目标为：

| 场景 | `Link` 的 `to` |
|------|----------------|
| project + 授权已同意（`done_agreed`） | `/consent?focusProtocolId={resolvedProtocolId}`（有 `protocol_id` 时）；否则 `/consent` |
| profile + 签名已保存（`done_profile`） | `/consent/witness-staff?focusWitnessStaffId={resolvedWitnessStaffId}` |

3. **名单**路径在点击前可 **`persistWitnessStaffListFocusId(id)`** 写入 `sessionStorage`，防止 Hash 下 query 解析失败时丢焦点（见 `witnessStaffListFocusStorage.ts`）。

## 3. 必须使用 Hash 路由并与「基路径」一致

本执行台使用 **`HashRouter`**（`App.tsx`），浏览器地址形态为：

```text
https://域名/execution/#/consent?focusProtocolId=123
```

- **路由路径**在 **`#` 后面**：`/consent`、`/consent/witness-staff`、`/witness-verify`。
- **`Link to="/consent?focusProtocolId=1"`** 会变为 **`#/consent?focusProtocolId=1`**，**不要**写成只打开 `https://域名/execution/` 而无 hash，否则会落在**默认重定向**（本仓库为 `#/dashboard`，表现为「回到执行台首页」）。

**Vite** 配置 `base: '/execution/'`（`vite.config.ts`）：部署后静态资源挂在 `/execution/`，但 **SPA 路由仍由 hash 决定**；邮件里的 **`{base}`** 必须与线上可打开的「执行台入口 URL 前缀」一致。

## 4. 知情管理列表：高亮 `focusProtocolId`

**页面**：`ConsentManagementPage.tsx`。

| 步骤 | 实现要点 |
|------|----------|
| 读参 | **`useSearchParams()`** 读取 **`focusProtocolId`**。 |
| 请求列表 | `consent-overview` 附带 **`focus_protocol_id`**，让**服务端返回该行所在页**的 `page`。 |
| 高亮 | 本地 state 标记行 id；表格行加背景样式。 |
| 清理 URL | 高亮后 **`replace: true`** 从 search 中 **删除 `focusProtocolId`**。 |
| 视图 | 保持**列表视图**，勿与「进入单项目」的 `protocolId` 深链混淆（后者会 `delete` query）。 |

## 5. 双签名单：高亮 `focusWitnessStaffId`

**页面**：`WitnessStaffPage.tsx`。

| 步骤 | 实现要点 |
|------|----------|
| 读参 | **`useSearchParams`** + `location.search` + **解析 `window.location.hash` 内 query**（`parseFocusWitnessStaffIdFromHash`）。 |
| 兜底 | **`location.state.focusWitnessStaffId`** + **`sessionStorage`**。 |
| 请求列表 | `listWitnessStaff` 传 **`focus_witness_staff_id`**，服务端返回所在 **page**。 |
| 高亮 | 行 class（如 `cnkis-witness-staff-focus`）+ **`scrollIntoView`**。 |
| 清理 | **`navigate` 去掉 search** + **`clearWitnessStaffListFocusStorage`**。 |

## 6. 飞书 / 统一登录：未登录时为何像「回首页」

若用户**未登录**就点「打开知情管理」，会先走 **OAuth**；回调后若**没有恢复原来的 hash**，会落到默认页（如 **dashboard**）。

**本仓库**：`@cn-kis/feishu-sdk` 的 `AuthResult` 含 **`postLoginRedirect`**，应恢复 **`#/consent?...`**（见 `packages/feishu-sdk/src/auth.ts`）。

## 7. 排查清单（「总是回到执行台首页」）

| # | 检查项 |
|---|--------|
| 1 | 完成页按钮是否为 **`Link`/`navigate` 到 `#/consent...`**，而不是 `<a href="/">` 或只打开站点根路径 |
| 2 | 生产环境 **`EXECUTION_PUBLIC_BASE_URL`** 是否与部署的 **执行台 URL** 一致 |
| 3 | 是否使用 **`HashRouter`**；若用 **`BrowserRouter`**，需保证服务器与 OAuth **不剥掉** query |
| 4 | 登录回调是否恢复 **postLoginRedirect / hash** |
| 5 | 名单页是否实现 **hash 内 query 多路解析** + **sessionStorage 兜底** |
