# Skill: resolve-conflict（解决 git merge 冲突）

## 用途

当 `sync-task` 或 `git merge` 遇到冲突时，帮助用户安全地解决冲突，不丢失任何一方的有效改动。

## 使用时机

- `./ops/scripts/dev-task.sh sync-task` 输出冲突提示
- 用户看到 `CONFLICT (content): Merge conflict in ...`
- 用户说「有冲突了，帮我解决」

## 执行步骤

### 第 1 步：了解冲突全貌

```bash
git status                   # 列出所有冲突文件（标记 UU）
git diff --diff-filter=U     # 查看冲突内容
```

### 第 2 步：逐文件分析

对每个冲突文件，分析：
- `<<<<<<< HEAD`（当前分支的内容）是什么改动
- `>>>>>>> origin/main`（main 的内容）是什么改动
- 两者是否有业务上的冲突（还是只是代码位置冲突）

### 第 3 步：确定解决策略

| 情况 | 策略 |
|------|------|
| 双方改动完全不同区域 | 保留两者，删除冲突标记 |
| main 是 bug 修复，当前是新功能 | 保留两者，通常可并排存在 |
| 双方修改了同一行逻辑 | 必须由用户（业务负责人）确认保留哪个版本 |
| main 重构了你正在改的文件 | 需要将当前改动适配到新结构 |

**重要**：对于业务逻辑层面的冲突，AI 不得自行决定"以哪边为准"，必须向用户说明两版差异并请求确认。

### 第 4 步：解决冲突

帮用户处理冲突文件（展示修改后的代码），确认后执行：

```bash
git add <已解决的文件>
git merge --continue
```

或者若决定放弃：

```bash
git merge --abort
```

### 第 5 步：验证解决结果

```bash
git status          # 确认无残余冲突
git log --oneline -3  # 确认合并提交已生成
```

## 业务同事操作指南

如果你是业务同事，遇到冲突时：

1. **不要慌**，冲突不会丢失任何代码，只是 Git 不知道选哪个版本
2. 把完整的冲突输出截图或粘贴给 Cursor，说：
   ```
   「我在同步 main 时出现了冲突，帮我解决，以 main 的最新版本为准」
   ```
   或：
   ```
   「我在同步 main 时出现了冲突，我的改动是 xxx，main 的改动是 xxx，
   我需要保留我的逻辑」
   ```
3. AI 会帮你分析并给出修改后的代码
4. 如果完全搞不定，运行 `git merge --abort` 取消合并，联系技术同事

## 注意事项

- ❌ 不要直接删除 `<<<<<<` 和 `>>>>>>` 之间的所有内容，这会丢失代码
- ❌ 不要在冲突未解决时提交（`git commit` 会失败）
- ✅ 每次解决冲突后都要运行 `git status` 确认干净
